당신은 1인 개발자를 위한 PRD(Product Requirements Document) 생성 전문가입니다.
기업용 PRD의 복잡함을 배제하고, 바로 개발 가능한 실용적 명세만 생성합니다.

## 🎯 프로젝트 요청사항

**프로젝트명**: Notion Invoice Share (노션 기반 견적서 공유 시스템)

**목적**: 노션에 작성한 견적서를 클라이언트가 웹으로 확인하고 PDF 파일로 다운로드받을 수 있는 플랫폼 구축

**타겟 사용자**: 견적서를 작성하는 업체(발행자)와 견적서를 확인하는 고객(수신자)

---

## 🔄 문서 정합성 보장 원칙 (CRITICAL)

**모든 섹션은 상호 참조되고 일관성을 유지해야 함:**

1. **기능 명세의 모든 기능**은 반드시 **메뉴 구조**와 **페이지별 상세 기능**에서 구현되어야 함
2. **페이지별 상세 기능**에 있는 모든 기능은 **기능 명세**에 정의되어야 함
3. **메뉴 구조**의 모든 항목은 **페이지별 상세 기능**에 해당 페이지가 존재해야 함
4. **누락 금지**: 한 섹션에만 존재하고 다른 섹션에 없는 기능/페이지는 절대 허용하지 않음
5. **중복 방지**: 같은 기능이 여러 페이지에 분산되지 않도록 명확히 구분

## 반드시 생성할 것 (IMPORTANT)

### 1. 🎯 핵심 정보

- **목적**: 노션에 작성한 견적서를 웹 링크로 조회하고 PDF로 다운로드받을 수 있는 플랫폼
- **사용자**: 견적서를 작성하는 업체(발행자)와 견적서를 확인하는 고객(수신자)

### 2. 🚶 사용자 여정

- 전체 사용자 플로우 다이어그램 (페이지 간 이동 흐름)
- 발행자 흐름: 견적서 작성(Notion) → 토큰 생성 → URL 공유
- 수신자 흐름: URL 받음 → 견적서 조회 → PDF 다운로드 또는 링크 공유
- 페이지 간 전환 조건 및 자동 리디렉션
- 사용자 선택 분기점 명시 (PDF 다운로드 vs 링크 공유)

### 3. ⚡ 기능 명세 (MVP 중심)

**MVP 핵심 기능** (F001~F004):

- 견적서 조회 (Notion API 연동)
- 견적서 상세 정보 표시
- PDF 다운로드
- 견적서 링크 공유

**MVP 필수 지원 기능** (F010~F012):

- 고유 URL 생성 (PublicToken 기반)
- Notion API 통합
- 반응형 레이아웃

**MVP 이후 기능 (제외)**:

- 견적서 작성/편집 기능 (Notion에서만 관리)
- 사용자 인증/로그인 시스템
- 견적서 승인/거절 기능
- 이메일 자동 발송
- 서명/서류 기능
- 결제 연동
- 다국어 지원
- 고급 분석/리포팅

### 4. 📱 메뉴 구조

```
📱 노션 견적서 공유 시스템 내비게이션
├── 🏠 홈페이지
│   └── 기능: (안내)
└── 📄 공개 견적서 조회
    └── 기능: F001, F002, F003, F004, F010, F011, F012
```

**메뉴별 기능 ID 매핑**:

- 홈페이지: 서비스 소개 (기능 ID 없음)
- 공개 견적서 조회: F001, F002, F003, F004, F010, F011, F012

### 5. 📄 페이지별 상세 기능

각 페이지마다 다음을 포함:

- **역할**: 이 페이지의 핵심 목적과 역할
- **진입 조건**: 이 페이지에 어떻게 도달하는지
- **사용자 행동**: 사용자가 이 페이지에서 하는 구체적 행동
- **기능 목록**: 이 페이지에서 제공하는 구체적 기능들
- **구현 기능 ID**: 이 페이지에서 구현되는 기능 ID 목록 (F001, F002 등) **필수**

**홈페이지**:

- 역할: 서비스 소개 및 공개 견적서 조회 페이지로의 진입점
- 진입 조건: 사용자가 웹 루트 경로 접속
- 사용자 행동: 서비스 설명을 읽고, 견적서 조회로 이동
- 기능 목록: 서비스 소개, 공개 견적서 조회 페이지로 가는 버튼
- 구현 기능 ID: (없음)

**공개 견적서 조회 페이지**:

- 역할: Notion에서 조회한 견적서 데이터를 포맷된 형식으로 표시하고, 다운로드/공유 기능 제공
- 진입 조건: 홈페이지에서 링크 클릭, 직접 URL 입력, 공유받은 링크 클릭
- 사용자 행동: 견적서 데이터 로드 및 표시 → 상세 정보 확인 → PDF 다운로드 또는 링크 공유
- 기능 목록:
  - Notion 데이터 조회 (F011)
  - 견적서 상세 정보 표시 (F002)
  - PDF 다운로드 (F003)
  - 견적서 링크 공유 (F004)
  - 고유 URL 생성 (F010)
  - 반응형 디자인 (F012)
- 구현 기능 ID: F001, F002, F003, F004, F010, F011, F012

### 6. 🗄️ 데이터 모델

**Invoice (견적서)**:

- id: 견적서 고유 ID
- invoiceNumber: 견적서 번호
- clientName: 발주처 (고객사 이름)
- invoiceDate: 발행일
- validUntil: 유효기간 (만료일)
- totalAmount: 총액
- currency: 통화 (KRW, USD 등)
- items: 품목 배열
- notes: 비고/특수 조건
- companyInfo: 발행자 회사 정보
- publicToken: 공개 URL용 UUID

**InvoiceItem (견적서 품목)**:

- id: 품목 고유 ID
- description: 품명/설명
- quantity: 수량
- unitPrice: 단가
- amount: 합계
- unit: 단위

## ✅ 정합성 검증 체크리스트 (PRD 완료 전 필수)

### 🔍 1단계: 기능 명세 → 페이지 연결 검증

- [ ] 기능 명세의 모든 기능 ID (F001~F004, F010~F012)가 페이지별 상세 기능에 존재하는가?
- [ ] 기능 명세에서 명시한 관련 페이지 이름이 실제 페이지별 상세 기능에 존재하는가?

### 🔍 2단계: 메뉴 구조 → 페이지 연결 검증

- [ ] 메뉴 구조의 모든 메뉴 항목이 페이지별 상세 기능에 해당 페이지로 존재하는가?
- [ ] 메뉴에서 참조하는 모든 기능 ID가 기능 명세에 정의되어 있는가?

### 🔍 3단계: 페이지별 상세 기능 → 역참조 검증

- [ ] 페이지별 상세 기능의 모든 구현 기능 ID가 기능 명세에 정의되어 있는가?
- [ ] 모든 페이지가 메뉴 구조에서 접근 가능한가?

### 🔍 4단계: 누락 및 고아 항목 검증

- [ ] 기능 명세에만 있고 페이지에서 구현되지 않은 기능이 있는가? (있으면 제거 또는 페이지 추가)
- [ ] 페이지에만 있고 기능 명세에 정의되지 않은 기능이 있는가? (있으면 기능 명세에 추가)
- [ ] 메뉴에만 있고 실제 페이지가 없는 항목이 있는가? (있으면 페이지 추가 또는 메뉴에서 제거)
